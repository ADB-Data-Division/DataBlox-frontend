datablox_od.tourism#

Module contents#

Provides functions for inferring tourism activities.

datablox_od.tourism.identify_tourism_trips(trips, coordinates_column, arrival_time_column, departure_time_column, administrative_region_column, trip_history_length_in_months=2, max_frequency_in_trip_history=1, max_trip_duration_in_days=30, min_distance_from_origin_in_km=30, verbose=False)#

Identifies which trips are tourism trips.

Important: This function assumes that the trips come from only a single device.

We refer to the United Nations’ International Recommendations for Tourism Statistics 2008 [1] for the elements in defining a tourism trip (paragraphs 2.52 and 2.53):

  • It should not be a frequently made trip.

    To this end, we consider the device’s history of past trips. We construct an observation window consisting of trip_history_length_in_months months up to and including the trip’s start date. If the administrative regions (based on administrative_region_column) visited in the trip of interest form a subset of the administrative regions visited in more than max_frequency_in_trip_history other trips that started within the observation window, then the trip of interest is not considered a tourism trip.

    To illustrate, suppose the trip of interest started on May 1, 2020, and consists of visits to administrative regions A, B, and C. Let trip_history_length_in_months be 2; this means that the observation window is from March 1, 2020, to May 1, 2020. If there exist more than max_frequency_in_trip_history other trips that started within this observation window and also involve visiting A, B, and C (and possibly more regions), then the trip of interest is not considered a tourism trip.

  • Its duration should be at most max_trip_duration_in_days.

    A trip’s start time is the time when the device departed from its usual residence. A trip’s end time is the time when the device arrived back at its usual residence.

  • It should involve crossing administrative borders.

    In other words, under the administrative_region_column, there should be the administrative region of the device’s usual residence and at least one other administrative region.

  • It should involve traveling at least min_distance_from_origin_in_km from the device’s usual place of residence. This distance requirement can be useful in filtering the trips of devices residing very close to administrative borders.

    We treat the starting point of the trip as a proxy for the usual place of residence.

Parameters:

  • trips (geopandas.geodataframe.GeoDataFrame) –

    Contains the trips from the device of interest. The trips should come from only a single device. At the minimum, this data frame should have columns for the:

    • Coordinates of the staypoints. The coordinates of each staypoint

    • Arrival times at the staypoints. The arrival time at each staypoint should be stored as a numpy.datetime64 object, which is the return type when pandas.to_datetime() is used.

    • Departure times at the staypoints. The departure time from each staypoint should be stored as a numpy.datetime64 object, which is the return type when pandas.to_datetime() is used.

    • Administrative regions where the staypoints are located.

  • coordinates_column (str) – Name of the column containing the coordinates.

  • arrival_time_column (str) – Name of the column containing the arrival times.

  • departure_time_column (str) – Name of the column containing the departure times.

  • administrative_region_column (str) – Name of the column containing the administrative regions.

  • trip_history_length_in_months (int) – Number of past months to be considered in computing the trip frequency.

  • max_frequency_in_trip_history (int) – Maximum trip frequency to be considered a tourism trip.

  • max_trip_duration_in_days (float) – Maximum trip duration (in days) to be considered a tourism trip.

  • min_distance_from_origin_in_km (float) – Minimum traveled distance (in km) from the trip’s starting point to be considered a tourism trip.

  • verbose (bool) – Whether to print progress messages to the standard output.

Return type:

tuple[geopandas.geodataframe.GeoDataFrame, pandas.core.frame.DataFrame]

Returns:

The first return value contains the tourism trips.

The second return value contains statistics on each tourism trip:

  • num_administrative_regions_visited – Number of distinct administrative regions visited. This count does not include the administrative region of the device’s usual residence.

  • trip_duration_in_days – Trip duration (in days).

  • farthest_distance_from_origin_in_km – Distance (in km) of the farthest staypoint from the trip’s starting point.

  • trip_frequency – Trip frequency. For example, a value of 2 means that, in the past trip_history_length_in_months months, the device made two other trips that involved visiting the same set of administrative regions visited in the tourism trip of interest (or a superset thereof).

References#

[1]

United Nations. International recommendations for tourism statistics 2008. https://unstats.un.org/unsd/publication/seriesm/seriesm_83rev1e.pdf. 2010.

datablox_od.tourism.identify_trips(staypoints, arrival_time_column, departure_time_column, residence_column, max_days_between_two_consecutive_staypoints=2, verbose=False)#

Identifies trips given a series of staypoints.

Important: This function assumes that the staypoints come from only a single device.

We refer to the United Nations’ International Recommendations for Tourism Statistics 2008 [2] for the definition of a trip (paragraph 2.7): “travel by a person from the time of departure from his usual residence until he/she returns”.

We introduce the additional criterion that every pair of (temporally) consecutive staypoints should be at most max_days_between_two_consecutive_staypoints apart. This is to parameterize the assumption that the device is still in the same trip when no pings were recorded for a possibly extended period of time.

Parameters:

  • staypoints (geopandas.geodataframe.GeoDataFrame | pandas.core.frame.DataFrame) –

    Contains the staypoints from the device of interest. The staypoints should come from only a single device. At the minimum, this data frame should have columns for the:

    • Arrival times at the staypoints. The arrival time at each staypoint should be stored as a numpy.datetime64 object, which is the return type when pandas.to_datetime() is used.

    • Departure times from the staypoints. The departure time from each staypoint should be stored as a numpy.datetime64 object, which is the return type when pandas.to_datetime() is used.

    • Whether the staypoints correspond to the device’s usual residence. This information should be stored as a Boolean value.

  • arrival_time_column (str) – Name of the column containing the arrival times.

  • departure_time_column (str) – Name of the column containing the departure times.

  • residence_column (str) – Name of the column containing information on whether the staypoints correspond to the device’s usual residence.

  • max_days_between_two_consecutive_staypoints (float) – Maximum number of days between every pair of (temporally) consecutive staypoints in a trip.

  • verbose (bool) – Whether to print progress messages to the standard output.

Return type:

geopandas.geodataframe.GeoDataFrame

Returns:

Trips.

References#

[2]

United Nations. International recommendations for tourism statistics 2008. https://unstats.un.org/unsd/publication/seriesm/seriesm_83rev1e.pdf. 2010.
